\section{Method}
\label{sec:method}

\subsection{Problem Formulation}

Given a user $u$ with interaction history $H_u$ and a candidate set
$C_u=\{c_1,\ldots,c_n\}$, the same-candidate reranking task is to order all
candidates so that the held-out next item is ranked as highly as possible. In
our formal setting, $n=101$ for every test event: one positive item and 100
negatives. Every compared method must produce one finite score per
$(u,c_i)$ pair under the same key schema.

The working scorer in \method{} is a task-grounded candidate relevance
posterior $p_i=P(y_i=1\mid H_u,c_i)$: a pointwise relevance probability that
the LLM emits when asked to judge each candidate under the next-interaction
task. Around this posterior we define an \emph{analyzed design space} of
uncertainty-adjusted ranking. We attach to each posterior an uncertainty
estimate $U_i\in[0,1]$ and define a one-parameter ranking family
\begin{equation}
    r_i = p_i(1-U_i)^\eta ,
    \label{eq:risk-score}
\end{equation}
where the risk exponent $\eta$ is fixed or selected using validation data only.
When $\eta=0$, the score reduces to the bare relevance posterior. We treat this
family as a hypothesis to be tested rather than as validated machinery: the
ablations in Section~\ref{sec:results} ask whether any positive $\eta$ or any
uncertainty term improves same-candidate ranking, and we report that they do
not. The $\eta=0$ row is test-best in all four domains, so the relevance
posterior itself is the operative ranking signal and the risk term is retained
only as an analyzed, and empirically inert, design choice.

The formulation is intentionally limited to reranking. It assumes a candidate
set has already been generated and asks whether a pointwise language-model
relevance posterior orders the same candidates more reliably than existing
LLM-based scoring and representation baselines. This choice gives the method a
narrow but auditable contract: for every event, each candidate must receive
exactly one posterior, one uncertainty value, and one final score before the
shared importer computes metrics.

\subsection{Task-Grounded Candidate Evidence}

For each user-candidate pair, the LLM is asked to evaluate the candidate under
the next-interaction task rather than under a generic similarity task. The
signal row records a pointwise relevance probability, a model-reported adjusted
probability, evidence support, counterevidence strength, and a short rationale.
The relevance probability is the working ranking score; the remaining fields
feed the analyzed uncertainty design space. The prompt does not reveal the
positive item index or test label.

The signal package is reproducible at the artifact level. Provenance records
the generation command, split-specific task paths, parser-compatible schema,
row counts, parse-failure count, and hashes for the signal and candidate
files. Rows with invalid probability, support, counterevidence, or key fields
are rejected before score export. The formal packages used for the paper pass
with 10,000 events, 1,010,000 candidate rows, exact candidate-key coverage, and
zero parse failures per domain.

The concrete signal-row runner fixes one parser-compatible schema and requests
only JSON fields:
\begin{center}
\small
\begin{tabular}{@{}ll@{}}
schema & \texttt{ccrp\_v3\_signal\_rows.2026-06-06}\\
field & \texttt{relevance\_probability}\\
field & \texttt{calibrated\_relevance\_probability}\\
field & \texttt{evidence\_support}\\
field & \texttt{counterevidence\_strength}\\
field & \texttt{reason}
\end{tabular}
\end{center}
The parser first accepts strict JSON or a JSON-object substring, then
field-level numeric fallbacks; fallback rows are marked through the
parse-success flag. Paper packages have zero parse failures under a maximum
allowed rate of 0.005. Generation provenance records the Qwen3-8B vLLM path,
temperature 0.1, 120 maximum new tokens, 5,000 users per chunk, and 101
expected candidates per event.

This evidence format separates three roles that are often conflated in LLM
recommendation: the model's relevance estimate, the support for that estimate,
and the amount of unresolved risk. The final score exporter keeps all candidate
keys aligned with the same-candidate task so downstream metrics can be audited
without relying on candidate order or missing-score imputation.

Let $\hat{p}_i$ denote the pointwise relevance probability emitted by the LLM
evidence parser, and let $p_i$ denote the schema field
\texttt{calibrated\_relevance\_\allowbreak{}probability} consumed by the
paper-facing scorer. We emphasize that this field is a model-reported adjusted
probability, not the output of a fitted statistical calibrator: \method{} does
\emph{not} estimate a post-hoc calibration map, and we make no
reliability-diagram or expected-calibration-error guarantee about it. In the
paper-facing configuration the working ranking score is the pointwise
posterior; validation controls only select among the score family, weights, and
$\eta$, and the scorer records the gap $|\hat{p}_i-p_i|$ as one input to the
uncertainty design space. If the adjusted field is absent, the exporter falls
back to $\hat{p}_i$ and the provenance exposes that source choice; in practice
the two probabilities track each other closely, which is consistent with the
ablation finding that the calibration-gap term carries little ranking signal.
For candidate evidence we also record support $s_i\in[0,1]$ and counterevidence
$q_i\in[0,1]$. The effective evidence score is
\begin{equation}
    e_i = \max(0, \min(1, s_i-q_i)).
\end{equation}
This places counterevidence inside the evidence channel rather than as a
separate post-hoc penalty. The leave-one-component-out diagnostics later test
whether this channel adds ranking signal under the frozen same-candidate
configuration, and find that it does not.

\subsection{Uncertainty Decomposition}

As part of the analyzed design space, \method{} derives an event-level and
candidate-level uncertainty estimate from four signals. We define these
components precisely so that each can be ablated; we do not assume any of them
improves ranking, and our experiments show that they do not.

\begin{itemize}
    \item \textbf{Boundary ambiguity:} candidates close to the decision
    boundary are treated as less certain than clearly high- or low-relevance
    candidates.
    \item \textbf{Calibration gap:} disagreement between the pointwise
    relevance probability and the model-reported adjusted probability, used as a
    proxy for posterior instability (not a measured calibration error).
    \item \textbf{Evidence insufficiency:} weak support from user history and
    item evidence increases uncertainty.
    \item \textbf{Counterevidence:} opposing evidence raises the risk attached
    to an otherwise high score.
\end{itemize}

These components are deliberately auditable. Section~\ref{sec:results}
reports leave-one-component-out diagnostics rather than assuming that any
component is beneficial. The resulting evidence is the basis for a
\emph{characterized negative result}: boundary uncertainty is inert in all four
domains, removing counterevidence and the risk penalty is non-worse, and a
confidence-only variant matches or exceeds the full configuration. The
decomposition is therefore reported as an analyzed but empirically inert design
space, not as necessary machinery.

In the current implementation, boundary ambiguity is computed as
$b_i=4p_i(1-p_i)$, which is largest near the posterior boundary and smallest
near 0 or 1. The calibration-gap term is $g_i=|\hat{p}_i-p_i|$. Evidence
insufficiency is $1-e_i$. With a validation-controlled weight triple
$w=(w_b,w_g,w_e)$, normalized to sum to one, the candidate uncertainty is
\begin{equation}
    U_i = \operatorname{clip}_{[0,1]}
    \left(w_b b_i + w_g g_i + w_e(1-e_i)\right).
    \label{eq:uncertainty}
\end{equation}
The default/preregistered weight triple is $(0.5,0.3,0.2)$, and the
hyperparameter module reports validation/test sensitivity over the allowed
weight grid. The counterevidence ablation sets $q_i=0$ before computing
$e_i$; the boundary, calibration-gap, and evidence-support ablations remove
one uncertainty term and renormalize the remaining uncertainty weights. This
renormalization is why we interpret ablations as diagnostic redistribution
tests, not as proofs of independent causal necessity.

\subsection{Ranking Family and Risk Term}

The ranking-family score in Equation~\ref{eq:risk-score} is multiplicative:
uncertainty would discount the base relevance posterior through $(1-U_i)^\eta$.
This is different from subtracting a fixed uncertainty penalty, and it matters
for claim interpretation. In principle a high-probability but high-uncertainty
candidate could be damped while a low-probability candidate cannot become
attractive only because it is certain. Empirically, however, the
discounting does not help: the test-best exponent is $\eta=0$ in every domain,
which collapses Equation~\ref{eq:risk-score} to the bare posterior $r_i=p_i$.
We therefore treat the multiplicative form as the analyzed hypothesis and the
$\eta=0$ posterior as the operative scorer. When the risk-penalty ablation is
active, or when a diagnostic score mode does not use uncertainty, the score is
the posterior or evidence score directly.

For each event, candidates are sorted by descending final score. Ties
are broken by a deterministic hash keyed by source event, user, item, and a
fixed seed; this removes any dependence on original candidate order while
preserving exact reproducibility. The score exporter emits only the required
same-candidate schema $(\texttt{source\_event\_id},\texttt{user\_id},
\texttt{item\_id},\texttt{score})$ for the final metric importer.

\subsection{Validation Discipline}

All method choices that can affect the final score are frozen or selected
before test evaluation. Validation rows choose among allowed score modes,
weights, and $\eta$ settings when a validation-selected variant is used; the
current paper-facing comparison uses the preregistered full C-CRP setting and
reports validation/test sweeps as sensitivity evidence. Test rows are used only
for final metric reporting, paired tests, observation figures, component
diagnostics, and hyperparameter sensitivity plots. The test hyperparameter
sweeps reported in the paper are descriptive stability checks and cannot change
the method.

The paper-facing selection space is deliberately small. The main score mode is
\texttt{full}; diagnostic-only score modes are excluded from the main table.
The reported sensitivity grid is:
\[
\begin{aligned}
\eta &\in \{0,0.25,0.5,1,2,4\},\\
w &\in \{(0.5,0.3,0.2),(0.7,0.2,0.1),(0.4,0.4,0.2),\\
  &\qquad (0.4,0.2,0.4),(0.33,0.33,0.34)\}.
\end{aligned}
\]
The \texttt{confidence\_weight} control is kept diagnostic-only under
\texttt{confidence\_plus\_evidence} and cannot be used to select the full
C-CRP row.

The fail-closed implementation checks this discipline mechanically. A score
file is rejected if it has missing candidate keys, extra keys, duplicate keys,
non-finite scores, excessive fully-constant events, or evidence rows that do
not align with the same-candidate task. These checks are part of the method
contract because they prevent a superficially plausible uncertainty score from
entering the paper table without exact candidate coverage.
